### #41 — clean‑shutdown verify skip
`Shutdown()` writes `pblocktree->WriteFlag("nedb_clean_shutdown", true)` after the final chainstate flush, before `pblocktree.reset()`. Startup (`AppInitMain`, ~`init.cpp:1671`) reads the flag, immediately clears it, and only runs `pblocktree->Verify()` when `want_verify = !fReset && !fReindex && !clean_shutdown`. Clean last exit ⇒ trust the store, skip the scan. Unclean kill ⇒ verify (and wipe+resync on corruption).

### #42 — the big one: warm‑boot tip was the *header* tip
Root cause: `CBlockTreeDB::WriteBatchSync` (`txdb.cpp:296`) persisted the highest‑`nChainWork` entry in the dirty block‑index batch as `DB_TIP_HASH`. During headers‑first IBD the dirty batch includes **header‑only** indices (headers race thousands of blocks ahead of the connected chain), so the persisted "tip" was a header with no block body and no UTXO. On restart the node warm‑booted to that header height while the chainstate sat at the real connected height → `ReplayBlocks(): reorganization to unknown block requested` → abort; and `-reindex-chainstate` hung because warm boot ran, wiped the chainstate, and `ThreadImport` skipped the `ActivateBestChain` rebuild.

Three coordinated fixes:
1. `txdb.cpp WriteBatchSync` — only consider blocks with `BLOCK_HAVE_DATA && HaveTxsDownloaded()` for the persisted tip. Header‑only entries ignored.
2. `validation.cpp TryWarmBoot` — reject a persisted tip lacking `BLOCK_HAVE_DATA` and fall back to full scan. **Self‑heals an already‑poisoned on‑disk tip with no resync** (keyed on `nStatus` only — `nChainTx` isn't persisted and is still 0 on the warm‑boot‑loaded tip there).
3. `init.cpp` — don't warm‑boot under `-reindex-chainstate` (`!warmboot_blocked && !fReindex && !fReindexChainState`), so the wiped UTXO actually rebuilds.

### #43 — `-flushwindow=<n>` bounds the shutdown flush
The chainstate full‑flush only fired at ~90% dbcache / every 24h / at shutdown. On a young chain with a big dbcache the dirty UTXO accumulated for the whole session, so the single shutdown flush took minutes (looked like a hang; tempted hard kills). `-flushwindow=<n>` adds a trigger in `CChainState::FlushStateToDisk` (`validation.cpp:~2428`): force a full flush once `m_chain.Height() - nLastFlushHeight >= n`. Runs through the **existing** full‑flush path so block‑index tip and chainstate stay consistent (no half‑flush — half‑flush is what caused #42). Default `0` (off). Recommended dev value: `-flushwindow=1000`. **Validated live:** 2403‑coin flush in 0.93s, 3016‑coin in 2.41s, clean stop 473 coins in 0.25s.

## Architecture (one‑paragraph recap)

ITC C++ consensus layer (P2P, PoW, ITSL, scripts) is **untouched**. The storage seam `CDBWrapper` (`src/dbwrapper_nedb.cpp`) routes block‑index + chainstate (+ a `tokens` store) through the `nedb-ffi` C API (`nedb-ffi/src/lib.rs`) into `nedb_core_v2::Db` — a content‑addressed BLAKE2b Merkle DAG (MVCC `AS OF`, causal `caused_by`, tamper‑evident). Block **bodies** still live in flat `blk*.dat` files (classic Core). Warm boot ("Proof‑of‑Prefix") = verify local integrity + load a 2016‑block window + confirm the tip is canonical against `seed.interchained.org:17101`.

**Key engine fact (verified, `nedb-v2/src/store.rs:145`):** every object read **re‑hashes and bails on mismatch** ("object tampered"). So content‑addressed reads self‑verify — corruption can never be silently served; it fails loud on access.

## Open work (prioritized)

### 1. Verify‑narrowing — kill the 54‑minute scan at the root (itcd‑only, biggest win)
On an **unclean** boot the node re‑hashes the *entire* store via `nedb_verify` → `Db::verify` (`store.rs:207`). Measured at **3,266,626 ms (54 min)** for 739k objects on Mark's iMac (~226 obj/s, I/O‑bound — the per‑object‑file read pattern). It's also misplaced: it verified NEDB fine while the actual corruption was in a flat `blk*.dat` body.

**Proposal (Mark likes it; needs his final nod on the tamper‑evidence tradeoff):** stop calling the full `Verify()`. Rely on (a) the warm‑start MANIFEST head the engine already validates, (b) the 2016‑window load (each read self‑verifies per `store.rs:145`), (c) read‑time self‑verification for anything touched later, and (d) the peer seam for canonicity. This mirrors Core's own `VerifyDB` (checks the last N blocks, not all history). Turns 54 min → milliseconds. Tradeoff: deep/cold corruption caught **lazily** on read instead of eagerly at boot — standard and safe, because reads can't return bad data.

### 2. Shutdown segfault (abort path)
On a failed init (e.g. VerifyDB hits a corrupt `blk` body → `AbortNode`), `Shutdown()` tears down a half‑initialized node and the token subsystem **lazily opens `g_token_db` during shutdown** (`src/wallet/token.cpp:767` / `:833`: `if (!g_token_db) g_token_db = new CDBWrapper(.../tokens…)`) — that's the `NEDB: opening database tokens` line appearing *after* `Shutdown: In progress`, followed by `zsh: segmentation fault`. Same family as the Ctrl+C ticker segfault fixed earlier, but on the abort path. **Clean stops do NOT segfault** (verified — tokens DB already open, just closes). Fix direction: guard the token lazy‑open against shutdown/abort state, and null‑guard the teardown so an early abort unwinds cleanly. **Need a crash report** (`~/Library/Logs/DiagnosticReports/interchainedd-*.ips`) to pin the exact frame.

### 3. Transaction / wallet validation (Mark is testing)
Base wallet is stock Core 0.21 (BDB/sqlite in `…/wallets`, **not** NEDB) → low risk; needs a synced chainstate for correct UTXO view. The **token/asset layer** (`g_token_ledger` + `tokens` NEDB DB, RPCs in `src/wallet/rpcwallet.cpp`) is the newest, least‑hardened code and already implicated in the segfault — **expect edges there.** Mark will send `sendtoaddress` + a token op result/log once synced.

### 4. The deeper ceiling (ENGINE — deferred)
Per‑object‑file I/O is the root of the slow verify and seek‑bound reads. A **packed/segment object store** (batch many objects per file) would make verify *and* flush fast and remove the ceiling. This is an engine change → off the table until Mark green‑lights touching `nedb`.

## Run / test / operate

```bash
# normal run (recommended dev flags)
./interchainedd -addnode=seed.interchained.org:17101 -dbcache=150 -flushwindow=1000

# clean stop (bounded flush, sets clean-shutdown flag → next boot skips verify)
interchained-cli stop      # let the flush finish; do NOT Ctrl+Z / kill

# recover a corrupt blk body / poisoned state (skips the 54-min verify, rebuilds from blk files)
./interchainedd ... -reindex
```

- `-dbcache=150` keeps the cache small (more frequent, smaller flushes). `-flushwindow=1000` bounds the shutdown flush to ~1000 blocks of UTXO. Both are dials.
- The verify is **uninterruptible** (single blocking FFI call) — Ctrl+C won't land until it finishes. Don't kill mid‑verify; it just restarts next boot.
